## 1. 产品一句话

> **一个本地优先的个人 Agent 桌面应用：自带任意大模型 API（BYOK，多 Provider），既能做编码任务、也能处理日常事务；每一步执行都可观测，并能用 Eval 自进化闭环把自己越用越准。**

电梯陈述：市面 AI 助手要么把 agent 内核当黑盒、要么锁死单一模型。Agent Y 把「多 Agent 编排 + 全链路可观测 + Eval 自进化闭环」这套工程内核做成**看得见、可换模型、可量化改进**的桌面产品——编码场景先点亮闭环，日常事务作为插件接入。

## 2. 目标与非目标

### 2.1 产品目标
- **G1 可用**：能真实帮作者完成编码任务（沙箱内改代码跑通测试）和日常事务（日程/邮件/文档/检索）。
- **G2 内核三件套**：多 Agent 编排、全链路可观测、Eval 自进化闭环，做成与场景解耦的底座。
- **G3 BYOK 多模型**：用户填自己的 API（多 Provider 原生适配），自由选模型；并能在同一任务集上**横向对比不同模型**。
- **G4 桌面 GUI**：把执行轨迹、Eval 提升曲线、模型对比**可视化**，演示力强。

### 2.2 学习目标（贯穿指北针，与产品目标并列）
- **L1** 学会做一个真项目（结构 / 测试 / 可观测 / 迭代的工程化）。
- **L2** 吃透 Agent 内核：手写 agent loop、工具/技能体系、多 Agent 编排、Eval 自进化。
- 取舍原则：**能学到本质 + 工程立得住 > 最快出 Demo**；核心 loop 要能看懂、能改、能讲清楚。

### 2.3 非目标（v1 明确不做）
- 不做多用户 / 账号体系 / 云端托管（**本地优先**，单用户）。
- 不做模型托管/自训练；只做**调用与编排**。
- 不做插件市场、团队协作、移动端。
- 不追求覆盖所有 OS 的像素级原生；**v1 锁定 macOS**（架构跨平台友好，Win/Linux 后置）。
- **明确不抄的"重/险"功能**（即便 Codex/Marvis 有）：操控桌面 GUI 应用（Computer Use）、屏幕截屏式记忆（Codex Chronicle，连大厂都被批隐私）、自然语言操控系统设置/整机（Marvis 式 OS 控制）、跨设备手机控电脑、本地大模型全本地推理（硬件门槛高，仅作"本地 Provider"轻量支持）、6-agent swarm 编排。理由：技术与隐私成本高、危险操作多、偏离"编码优先"定位。

### 2.4 设计原则（本轮讨论敲定）
- **CLI 先行，渐进成 app**：先做命令行版把内核跑通，再做桌面 app；**app 内嵌终端（兼容 CLI）**，CLI 始终是一等公民。
- **"轻"指日常场景、不指核心体验**：刻意保持轻量的是**日常事务/个人助手场景**（不做大而全的待办/日程/生活管理）——这一侧核心就三样：提醒待办、把常做的事务沉淀成 skill、完善的 memory。但**核心 agent 体验要做精致**（审批分级、会话/线程、diff 审阅、记忆、实时 trace）——那是产品脊梁与简历价值，不能薄。详见 §4.1 体验外壳。
- **真能自用（dogfood）是硬指标**：作者要拿它当日常工具，所以**每个里程碑交付的都得"我自己当下就能用上"**，而非攒到最后才好用。每个 M 的验收都隐含"作者能用它干一件真事"。
- **产品定位：编码优先 + 轻量日常，对标 Codex 桌面端那一类（编码 agent + 精致桌面壳），不走腾讯 Marvis 的"操作系统级通用助手/自然语言操控整机"路线**——后者是另一物种且含大量危险系统操作，超出 v1 且偏离定位。
- **核心解耦 GUI**：后端是本地服务，CLI / 桌面 app / 未来其它外壳都只是它的客户端。

## 3. 目标用户与场景

**主用户画像**：作者本人——会 Python、在学 Java、想做一个能放简历又真能用的个人 Agent。
**次要画像（潜在）**：和作者类似的开发者/技术求职者，想要一个可换模型、能看清 agent 在干什么的本地助手。

**核心场景**
1. **编码任务（先做）**：「让这个仓库的失败测试通过」「按描述实现某函数」——agent 在 Docker 沙箱里读/改文件、跑测试，直到转绿；全过程在右侧 trace 可见。
2. **日常事务（后续插件）**：「整理今天日程并起草确认邮件」「在我的文档里找 X 并总结」——通过场景插件接入工具。
3. **自进化运维**：跑一遍编码任务集出基线 → 让系统据失败自动改进 → 看提升曲线和可解释改进记录。
4. **换模型对比**：同一任务集分别用 Claude / OpenAI / 本地模型跑，看通过率/成本/延迟对比。

## 4. 差异化价值主张

| 维度 | 主流 AI 助手 | Agent Y |
|---|---|---|
| Agent 内核 | 黑盒 | **可观测**：每步 LLM/工具/决策/token/延迟可视化 |
| 模型 | 锁死 1 家 | **BYOK 多 Provider 原生适配 + 模型横向对比** |
| 质量演进 | 靠人工调 prompt | **Eval 自进化闭环**：据失败自动改、仅当提升才保留、出量化曲线 |
| 形态 | 多为云端/网页 | **本地优先桌面 app**，密钥与数据不出本机 |
| 扩展 | 固定能力 | **场景插件化**：加能力 = 写插件，不动内核 |

### 4.1 体验外壳（借鉴 Codex 桌面端，分三档落地）

研究了 Codex 桌面端与腾讯 Marvis 后，把"让它真像个好用的助手"的功能拆成三档——**核心 agent 体验做精致，日常场景保持轻**：

**① 现在就纳入（便宜 + 高价值 + 已有地基）**
- **审批 / 沙箱分级 + 默认断网**：只读 / 问一下 / 自动 / 完全放开四档，网络默认关，危险操作要确认。Codex 最该抄的"信任骨架"，且 Claude Code 的 `allow/deny/ask` 是现成模板（见 `docs/code-study-cc.md §3`）。→ 落 F8.4，从"安全条目"升格为**用户可见的信任开关**，M1 起就有最小版。
- **会话持久化 + 历史 + 按项目/任务分线程**：会话不随关窗丢失，多件事分线程互不污染上下文。→ 落 F2.5 + F7.4，M2 随 GUI 落地。
- **实时 trace 可视化**：把 agent 在干什么可视化（Marvis 用"办公室"动画、Codex 用指挥中心）——**这本就是 Agent Y 三件套核心**（F3.3），强化即可。

**② 后置（post-MVP，做实"助手感"，对应 F6）**
- **定时自动化 + 待办/review 队列**（Codex Automations）：让常做的活按计划在后台跑、结果进队列等审——把 F6.1"提醒待办"做成真功能。
- **diff 审阅闭环**：GUI 里看/评论/手改 agent 的改动、一键跑测试——编码场景价值最高。→ 落 F5 + F7，M2–M3。
- **本地 vs 云端开关**：在 BYOK 上加一层隐私选项（用本地 Ollama 还是云端）。注意只做"选本地 Provider"，不承诺全本地大模型推理（硬件门槛高）。

**③ 明确不抄**：见 §2.3 非目标末条（Computer Use / Chronicle 屏幕记忆 / OS 操控 / 跨设备 / swarm）。

## 5. 功能需求

> 优先级：**P0** = MVP 必备（对应 M1–M4）；**P1** = 第二阶段（M5–M6）；**P2** = 远期。
> 每条给出用户故事 + 验收标准（节选关键项）。

### F1. 模型与 Provider 管理（BYOK） — P0
- **F1.1 Provider 适配（BYOK）** [P0]：统一 `LLMProvider` 接口（chat/stream、工具调用、tool_result、stop_reason、usage/token、thinking 归一化）。范围**简化为 Claude 原生 + OpenAI 兼容端点**——一个 OpenAI 兼容适配器即覆盖 **GPT / DeepSeek / 本地 Ollama·LM Studio / 多数云厂商**（用户填 base_url + key 即可）。
  - 交付节奏：**Anthropic 原生（M1）→ OpenAI 兼容适配器（M3）**；Gemini 等若不兼容 OpenAI，走其 OpenAI-兼容模式或后置。
  - 验收：新增一个端点 = 配置 base_url + key，不改 loop/可观测/eval 内核。
- **F1.2 连接管理（BYOK）** [P0]：用户故事——"作为用户，我能添加一个连接（选 Provider + 填 API key + 可选 base_url），保存后在模型下拉里选到它的模型。"
  - 验收：key 经 **OS keychain 加密存储**（macOS Keychain，via `keyring`），明文不落盘、不进日志；删除连接即清密钥。
- **F1.3 模型选择与能力声明** [P0]：每个模型带能力元数据（是否支持 tool use / thinking / 上下文窗口 / 价格）。
  - 验收：UI 标注模型能力；Runtime 据能力**优雅降级**（如非 Claude 不发 thinking/effort 参数）。
- **F1.4 按角色配模型** [P1]：orchestrator / subagent / eval-judge 可分别指定模型（如主力 Claude Opus 4.8、跑量 Sonnet 4.6、廉价子任务 Haiku 4.5 或本地模型）。

> Claude 调用约定（Anthropic 适配器）：`thinking:{type:"adaptive"}` + `output_config:{effort:"high"}`；不发 `budget_tokens`/`temperature`/`top_p`；大 `max_tokens` 走 streaming。model id：`claude-opus-4-8`($5/$25,1M) / `claude-sonnet-4-6`($3/$15,1M) / `claude-haiku-4-5`($1/$5,200K)。

### F2. Agent Runtime 核心 — P0
- **F2.1 手写 Agent Loop** [P0]：上下文拼装 → 模型调用(带工具) → 检测 tool_use → 执行工具 → 回灌 tool_result → **以"本轮有无 tool_use 块"决定续/止**（无工具调用即完成；不只看 stop_reason——code-study 与 OpenAI 文档双重佐证）；叠加 max_turns/预算硬上限防死循环。
  - 验收：能连续多步用工具完成一个任务，每步可被 tracer 记录。
- **F2.2 统一 Tool 插件接口** [P0]：name / description / input schema(Pydantic) / readOnly 并行提示 / execute(input, ctx)。注册表 + 按名分发。
- **F2.3 内置工具** [P0]：bash、read_file、write_file、edit_file（编码场景所需最小集）。
- **F2.4 子 Agent / 编排** [P1]：orchestrator 可派生 subagent（隔离子上下文、独立工具集、结果回传父级）。
- **F2.5 记忆 / 会话** [P0 最小 / P1 完善]：MVP 含最小会话持久化；**完善的 memory 系统**（跨会话长期记忆、按相关性召回、供自进化写 few-shot）列为**重点 P1**——这是"轻量个人助手"体验的关键，参考 `cc-resourcecode/memdir/`。

### F3. 可观测（Tracing） — P0
- **F3.1 执行轨迹采集** [P0]：每次运行产出一棵 span 树（run → 每步 LLM 调用 / 工具调用 / 决策），记录输入输出摘要、token、延迟、父子关系。
- **F3.2 Langfuse 接入** [P0]：trace 落 Langfuse（自托管）；tracer 接口抽象，可切换 console/OTel。
- **F3.3 GUI 内 trace 可视化** [P0]：右侧面板实时显示执行轨迹（见 F7）。
  - 验收：跑一个任务后，能在 GUI 和 Langfuse 两处回看完整调用树。

### F4. Eval + 自进化闭环（核心卖点） — P0
- **F4.1 Eval harness（编码）** [P0]：任务集（每个 = 最小 workspace + 测试 + 评分脚本），一键跑全集出通过率（pass@1 等）。
- **F4.2 自进化** [P0]：据失败聚类自动生成改进候选（改 prompt / 加 few-shot 记忆 / 调工具描述）→ 重跑 → **仅当通过率提升才保留**；产出**量化提升曲线 + 可解释改进记录 + 可回滚**。
- **F4.3 模型对比** [P1]：同一任务集跑多个模型，输出通过率/成本/延迟对比榜。
- **F4.4 事务类 Eval** [P1]：LLM-judge + 规则评分（半客观）。

### F5. 编码场景插件 — P0
- 注册编码工具集 + 提供编码 Eval 数据集；在 Docker 沙箱内执行。
- **F5.1 diff 审阅闭环** [P1]：GUI 里把 agent 的文件改动以 diff 呈现，可逐块查看 / 接受 / 拒绝 / 在编辑器打开手改，并一键跑测试（借鉴 Codex 桌面端，见 §4.1）。M2–M3 随 GUI 落地。
- 验收：给定"修复失败测试"任务，agent 在沙箱内完成并跑通测试；GUI 里能看 diff 再确认。

### F6. 轻量个人助手场景插件 — P1（MVP 之后，刻意保持轻）
- **F6.1 待办提醒**：记录 / 提醒待办事项（简单，不堆功能）。
- **F6.2 常用事务 → Skill**：把你经常处理的事务沉淀成可复用 skill（参考 `cc-resourcecode/skills/`：description + 何时用 + 工具集，按需加载）。
- **F6.3 依托 F2.5 的 memory**：助手"懂你"靠完善的记忆系统。
- **F6.4 定时自动化 + review 队列** [P1]：把常做的活配成"按计划/触发自动跑"，结果进一个待审队列等你收口（借鉴 Codex Automations，见 §4.1）。这是把 F6.1 从"提醒"升成"真能替你跑"的关键，但仍受审批分级(F8.4)约束、保持轻量。
- 验收：接入后**未改动 runtime/可观测/eval 内核**即跑通——证明换场景只写插件；整体体验**轻量**，不做重型 PIM/生活管理。
- 候选实现：待办/提醒后端可作为**独立 Spring Boot(Java) 服务**（见 §13），边界清晰、适合并行开发与简历。

### F7. 桌面 GUI — P0
- **F7.1 主界面（split view）** [P0]：左 Chat、右 Execution Trace（见 §8 线框）。
- **F7.2 模型/Provider 管理界面** [P0]：增删连接、填 key/endpoint、选默认模型、（P1）按角色配模型。
- **F7.3 Eval 看板** [P0/P1]：任务集通过率、提升曲线、（P1）模型对比榜。
- **F7.4 会话/线程 + 设置** [P0 最小 / P1 完善]：会话持久化与历史回看；**按项目/任务分线程切换**（多件事互不污染上下文，借鉴 Codex，见 §4.1）；偏好、沙箱配置、审批分级设置。
- **F7.5 打包为 macOS `.app`** [P0]：pywebview 套壳 + PyInstaller 出 `.app`。

### F8. 安全 / 隐私 / 沙箱 — P0
- **F8.1 密钥安全** [P0]：OS keychain 存储，绝不明文落盘/进日志/进 trace。
- **F8.2 代码执行沙箱** [P0]：编码任务在 **Docker 容器**内执行，限制网络/资源；不在宿主机直接跑模型生成的代码。
- **F8.3 本地优先** [P0]：会话、trace、配置默认存本机；不上传第三方（Langfuse 自托管在本地/自有机器）。
- **F8.4 审批 / 沙箱分级（信任开关）** [P0 最小 / P1 完善]：用户可见的信任分级，是核心 agent 体验的"骨架"（借鉴 Codex，见 §4.1）——
  - **审批模式**：只读 / 问一下（越权才停下确认，默认）/ 自动（不打断）/ 完全放开（仅可信环境）。
  - **沙箱模式**：只读 · 工作区可写（仓库内可改可跑、**网络默认关**）· 完全访问。
  - 工具按 `is_read_only` / `is_destructive` 自动归档；高风险/不可逆操作（删除、外发邮件、写仓库外）按所选模式提示确认。实现参考 `docs/code-study-cc.md §3` 的 `allow/deny/ask` 三态机制。
  - 交付节奏：M1 起最小版（只读自动放行 + 写操作确认），M2 起 GUI 暴露为开关，后续随自动化(F6.4)完善。

## 6. 非功能需求（NFR）
- **可扩展**：Provider、Tool、Scenario、Eval 数据集均为插件式接入。
- **性能**：单步交互流式输出；trace 采集不显著拖慢执行。
- **可靠**：模型/工具失败有重试与可读错误；沙箱超时可中断。
- **隐私**：默认全本地；任何外发（仅限用户选的 Provider）对用户透明。
- **可移植**：核心后端与 GUI 解耦（本地 HTTP/WS），换外壳（浏览器/Tauri/未来 SwiftUI）不动核心。
- **跨平台友好**：v1 锁 macOS，但不写死 macOS-only 依赖，便于后续 Win/Linux。

技术选型：Python(FastAPI) 后端 · `@anthropic-ai/sdk` 同级的 `anthropic` Python SDK + `openai` + `google-genai` 等原生 SDK · Pydantic（工具 schema）· Docker（沙箱）· Langfuse（可观测）· `keyring`（密钥）· 前端 React + pywebview 套壳。
研习参照（TS，仅作概念对照，不入库）：`cc-resourcecode/query.ts`(loop)、`Tool.ts`/`tools.ts`(工具)、`coordinator/`+`tools/AgentTool/`(子agent)、`memdir/`(记忆)。

## 10. 成功指标（Metrics）
- **闭环可量化**：自进化一轮后编码任务集通过率提升 ≥ X 个百分点（具体阈值 M3 定基线后定）。
- **多模型可用**：≥ 3 个 Provider（含 1 个本地）能在同一任务集跑通并对比。
- **可观测覆盖**：100% 的运行可在 GUI/Langfuse 回看完整 trace。
- **学习产出**：手写 loop 可对照 `cc-resourcecode/query.ts` 讲清差异（学习笔记入 `docs/`）。
- **可演示**：`.app` 双击即用，BYOK 后能完成一个编码任务并展示 trace + 提升曲线。
- **真能自用（dogfood）**：作者每个里程碑后都能拿当下版本干一件真事（M1 用 CLI 修个真 bug、M2 起在 app 里日常用）；目标——作者把它当**主力日常工具**而非只演示。

## 11. 风险与依赖
- **多 Provider 原生适配较重**：缓解——统一接口先定好，按 M1/M3/M5 分批加适配器；每家差异（工具调用格式、thinking、流式）在适配器内吸收。
- **前端学习成本**：作者 Python-first；缓解——前端用轻量方案，先把功能跑通再美化；后端/核心全在 Python。
- **沙箱安全**：模型生成代码必须只在 Docker 内执行，限制网络与资源。
- **密钥安全**：严格走 keychain，审计日志/trace 不含密钥。
- **范围发散**：编码场景先到 M4 闭环跑通，再开事务场景；GUI 先 split view，其余后置。

## 12. 决策记录（本轮敲定）/ 待定

**已敲定**
- **形态**：CLI 先行 → 桌面 app（内嵌终端兼容 CLI）。深度优先，内核先跑通。
- **Provider**：Claude 原生 + OpenAI 兼容端点（覆盖 GPT/DeepSeek/本地）。
- **前端**：由 AI 助手负责实现，作者按效果驱动修改。
- **助手定位 / 体验深浅**（本轮细化）：**日常事务场景刻意轻**（待办提醒 + 常用事务 skill + 完善 memory，列 MVP 之后）；**但核心 agent 体验要做精致**——审批分级、会话/线程、diff 审阅、记忆、实时 trace 这套"体验外壳"是产品脊梁，按 §4.1 三档分期落地。**"轻"≠"功能简单"。**
- **产品定位（本轮敲定）**：对标 **Codex 桌面端那一类（编码 agent + 精致桌面壳）**，借鉴其审批/会话/diff/记忆/自动化；**不走腾讯 Marvis 的"OS 级通用助手/自然语言操控整机"路线**（另一物种 + 危险操作多）。明确不抄 Computer Use / Chronicle 屏幕记忆 / OS 操控 / 跨设备 / swarm（见 §2.3）。
- **真能自用（dogfood）为硬指标**：每个里程碑交付都要"作者当下就能用上"，目标是把它当主力日常工具，不是只演示。
- **语言**：核心 Python；Java 找一个边界清晰模块承载（见 §13）。
- **交付形态**：**全程 Python，最终打包成 macOS `.app`/`.dmg`（pywebview + PyInstaller/Briefcase），不做语言重写**。前端是 Web 技术(JS)、核心是 Python，二者经本地 HTTP 接口通信——"前端 JS + 后端 Python + 打包成 app"即成品形态。曾讨论的"最后整体转 TS 编译"已否决：agent 瓶颈在 API 延迟不在宿主语言，"双击即用"靠打包而非语言；TS 单文件 CLI / VS Code 插件若将来需要，是独立的未来移植，靠 §架构 的语言无关接口契约保证可行，不作默认收尾步骤。
